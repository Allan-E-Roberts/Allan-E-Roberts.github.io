Pardon the formtting. This is an experimental feature I am still working on, and as such, uses very bad temporary code that I change often.

This page is pure HTML and CSS. No JavaScript, webscripts, nor even SCSS, SASS, nor SASSY was used. True: there is no official way to track mouse positions in css, nor any official way to track the progress of an animation, nor alter animations without resetting them. That doesn't mean such is impossible; it just means it takes more creativity.

There is still a problem with this tool: gimble lock. CSS was never designed with any intention of handling tasks such as this, and there is a bug with how it assigns coordinate reference frames. When you first refresh the page, you will notice that the mouse follows the cursor accurately when it's nearby, but after a while, only the top and bottom controls work as expected. An alterante way to view the bug is to locate the mouse diagonally from the icosahedron and follow the "home" tri. Notice that it follows a figure eight instead of making a full circle. CSS insistantly alters how references are assigned, and it wants to treat X and Y rotations differently. If there was a method to process both the same way, it would be easy (for absolute frames, nothing would be needed, and for relative frames, I could just plug in the conversions to transform the rotations back to the absolute frame), but all of the methods I know of get overriden when CSS 'fixes' the references.
